# CloudX VDI 플랫폼 초기 구축 흐름 (추측)

코드 분석 + 실제 환경 데이터 기반 추측. 실제 구축 가이드 미확인.

## 추측 근거

1. admin-portal에 도메인 생성 UI 없음 (A1001은 읽기 전용)
2. SA 첫 로그인 시 `/initialized` 마법사로 리다이렉트 (`acct_sts_cd === 'U013IN'`)
3. `init` 스토어에 단계별 설정 로직 존재 (`POST /v1/resource/init`)
4. app-ms-resource의 목록 조회가 로컬 DB 기반 → 데이터 동기화 단계 필요
5. `ResourceComponent`에 `getTenantId()`, `deleteTenant()` 존재 → OpenStack Project와 로컬 DB 양방향 관리
6. Zone 조회 쿼리에서 `CV_TNT_M.ZONE_NM`으로 테넌트-Zone 매핑 확인 (코드 검증)
7. 실제 환경 데이터: Default Zone(`tnt_id` 있음, 호스트 2대) + public-zone(`tnt_id: null`, 호스트 없음)

## 논리 ↔ 물리 연결 구조 (코드 검증 기반)

```
논리적 구조 (Keystone)          연결 지점                    물리적 구조 (Nova)
─────────────────────    ─────────────────────    ─────────────────────
Domain
  └── Tenant ──────────── CV_TNT_M.ZONE_NM ──────── Zone (= Host Aggregate)
        │                 CV_TNT_M.HOST_TNT_YN        ├── Host (compute-01)
        │                                              └── Host (compute-02)
        │                                                    │
        └── VM (가상 PC) ─── 배치(스케줄링) ──────────────────┘

        │                                    [Neutron]
        ├── 가상 네트워크 → 서브넷 → Port → VM에 IP 할당
        │                    └── 라우터 → 물리 네트워크 → 외부
        │
        │                                    [Cinder]
        └── Volume (디스크) → Volume Type → NetApp Backend
```

OpenStack에는 테넌트→Zone 연결이 공식적으로 없음. CloudX가 `CV_TNT_M` 테이블로 자체 구현.

### 호스트 테넌트 모드 (코드 검증 기반)

`CV_TNT_M.HOST_TNT_YN` 값에 따라 Zone 조회 쿼리 분기:

| 모드 | `HOST_TNT_YN` | Zone 매핑 방식 | 용도 (추측) |
|------|-------------|--------------|-----------|
| 일반 | `N` | `CV_TNT_M.ZONE_NM`으로 이름 매핑 | 호스트 공유 가능, 자원 효율 중시 |
| 호스트 테넌트 | `Y` | `CP_ZONE_M.TNT_ID`로 직접 매핑 | 호스트 독점, 물리적 격리 (추측) |

"호스트 테넌트"는 OpenStack 공식 개념이 아닌 CloudX 자체 확장.

## 추측 흐름

### 1단계: OpenStack 인프라 구축

CloudX 이전에 OpenStack 클러스터가 먼저 존재해야 함.

```
- Keystone: Domain 생성 (보통 'default' 단일 도메인)
- Nova: Compute Node(Host) 설치, 하이퍼바이저 구성
- Neutron: Provider Network(물리 네트워크) 구성, 외부 네트워크 연결
- Cinder: Storage Backend(NetApp) 연결
- Glance: 기본 OS 이미지 등록
```

NetApp 사용 이유: 공유 스토리지라서 Live Migration/HA(Evacuate) 시 디스크 복사 불필요, 데이터 손실 없음.

### 2단계: CloudX 플랫폼 배포

```
- plat-ms-vid4o: OpenStack 접속 정보 설정
  → application.yaml에 Keystone endpoint, admin credential
  → context path: /VID4Openstack, port: 12105
- app-ms-resource: plat-ms-vid4o 연결 설정
  → vid4o.url 설정 (RestTemplate으로 호출, Feign 아님 — 코드 검증)
  → MariaDB 스키마 생성
- app-ms-operation: 정책/운영 DB 스키마 생성
- admin-portal / user-portal: 프론트엔드 배포
- API Gateway 설정
```

### 3단계: 초기 데이터 세팅 (추측)

```
- SA 초기 계정 INSERT (acct_sts_cd = 'U013IN', grp_typ_cd = 'U001SUP')
- 공통 코드 데이터 INSERT (A003, A006, A009, P004, P006, U001, U004, U013 등)
- SA/TA 메뉴 트리 데이터 INSERT (A prefix / T prefix)
- 기본 정책 데이터 INSERT
```

### 4단계: SA 첫 로그인 → `/initialized` 마법사 (코드 검증 완료)

9단계 순차 설정 (자동/수동 모드):

```
Step 0: 도메인 — OpenStack에서 도메인 정보 동기화 (수동, 모달)
Step 1: 호스트 — GET /v1/resource/hosts/admin?code=init
Step 2: Zone — GET /v1/resource/zones?code=init
Step 3: 이미지 — GET /v1/resource/images?code=init
Step 4: Flavor — GET /v1/resource/flavors?code=init
Step 5: 네트워크 — GET /v1/resource/networks?code=init&usg_use_cd=P004P (물리만)
Step 6: 서브넷 — code=init
Step 7: 볼륨 — code=init
Step 8: Proxy — GET /v1/resource/proxy/L4/admin + L7/admin
```

완료 후: `acct_sts_cd` → `U013NR`, 로그아웃 후 재로그인.
상세: [admin-portal/init/01-초기-설정-마법사.md](../admin-portal/init/01-초기-설정-마법사.md)

### 5단계: 테넌트 생성 + Zone 할당

```
SA: 테넌트 관리 (A0802) → 테넌트 생성
  → 테넌트명, URL ID, HOST_TNT_YN, CPU/메모리 제한값 설정
  → CV_TNT_M에 저장 + OpenStack Project 생성 (추측)
  → CV_TNT_M.ZONE_NM에 Zone 매핑 → 논리(테넌트) ↔ 물리(Zone) 연결 확립
```

### 6단계: 인프라 리소스 + 정책 구성

```
[SA] 템플릿 구성:
  → Flavor 생성 (가상 PC 사양: CPU/RAM/디스크 스펙 정의)

  → 골든 이미지 생성 (코드 검증 기반 추측):
    ※ admin-portal에 이미지 레코드(CT_IMG_M) 자체를 생성하는 UI 없음
    ※ 조회 화면(GoldenImage.vue): register() 메서드 존재하지만 호출 버튼 없음
    ※ 생성 화면(GoldenImageCreate.vue): 이미 존재하는 이미지에 대한 스냅샷 실행
    ※ 이미지 레코드 생성 경로 추측:
       - /initialized 마법사 Step 3에서 Glance → 로컬 DB 동기화 시 INSERT
       - 또는 백엔드 내부 로직 / DB 직접 INSERT

    골든 이미지 생성 화면의 "VM 목록"은 일반 사용자 VM이 아님:
    - 주 테이블: CT_IMG_M (이미지 마스터)
    - 조인: CT_TNT_IMG_L (테넌트-이미지 연결) → CV_VM_M (이미지 생성용 전용 VM)
    - 관리자가 소프트웨어를 설치하기 위한 "작업용 VM"

    실제 생성 흐름:
    1. CT_IMG_M에 이미지 레코드 존재 (생성 경로 미확인)
    2. CT_TNT_IMG_L로 "이미지 생성용 VM"이 연결됨
    3. "접속" 버튼 → skbvdi://로 전용 VM 접속 → 관리자가 OS/SW 직접 설치
    4. "전원 끄기" → VM 정지
    5. "생성" 버튼 → VM 스냅샷 → Glance 이미지 추출 (3초 폴링)
    6. "배포" 버튼 → 이미지를 다른 테넌트에서도 사용 가능하게 배포
    ※ 이미지 파일을 "업로드"하는 방식이 아님 — 전용 VM의 디스크를 스냅샷

  → 템플릿 생성: Flavor + 골든 이미지 조합 = 가상 PC 설계도

[SA] 스토리지: 볼륨 타입 확인 + 디스크 QoS 생성 → 볼륨 타입에 associate

[SA] 정책 구성 (상세 정책 → 허브 순서):
  ① 상세 정책(A0506) — 개별 정책 먼저 생성
     → 예외 네트워크 (A050601): 2차 인증 면제 IP 대역 그룹
     → 접근 차단 (A050602): 서비스별 네트워크 차단/허용 규칙
     → 블랙/화이트리스트 (A050604/05): 실행 차단/허용 SW 목록
     → URL 리다이렉션 금지 (A050606): 차단 URL 목록
     → 미라이선스 (A050607): 라이선스 미인증 VM 보안 수준
     → Metadata (A050608): VM 하드웨어 설정 (모니터, USB, SPICE)
     → USB 정책 (A050609): USB 유형/제조사/모델 마스터 + 허용/차단 규칙
     → 보안 그룹 (A050611): OpenStack Neutron Security Group 동기화

  ② 인증 정책 — SA 레벨 기본 템플릿 정의
     → 사용자 인증 (A0501): 포털 로그인 시 2차 인증
     → 관리자 인증 (A0502): 관리자 포털 로그인 시 2차 인증

  ③ 가상 PC 정책 (A0503) — 허브: ①의 정책들을 묶는 역할
     → SA는 자체 설정(2차 인증, 해상도, 워터마크, 세션 만료)만 편집
     → 접근 차단/URL Redirection/미라이선스/Metadata/Proxy 드롭다운은 SA에서 disabled
     → TA가 이 드롭다운에서 ①의 정책을 선택하여 VM에 적용

  ④ 백업/스냅샷 정책 (A0505): 자동 백업/스냅샷 스케줄 (매일/매주/매월)

[SA] RBAC: 메뉴/기능/API 정의 (A1208) + TA용 권한 그룹 생성 (T1202)
```

### 7단계: TA 계정 생성 및 서비스 운영

```
SA: TA 계정 생성 → 테넌트 + 권한 그룹 + 인증 정책 할당

TA 로그인 후:
  1. 네트워크 구성
     → 가상 네트워크 → 서브넷 (CIDR/DNS/DHCP) → 라우터 (물리↔가상 연결)

  2. 정책 커스터마이징 (SA 템플릿 기반)
     → 인증 정책(T0501/T0502): 테넌트별 로그인 제한 조건 커스터마이징
     → 가상 PC 정책(T0503): SA가 disabled 해둔 드롭다운 활성화
        → 접근 차단/URL Redirection/미라이선스/Metadata/Proxy 정책 선택
        → USB 정책 선택 (보안 항목 테이블에서)
        → 보안 그룹 선택
     → 백업/스냅샷 정책(T0505): 테넌트별 스케줄 커스터마이징

  3. VM 프로비저닝
     → 가상 PC 그룹 (템플릿 + 네트워크) → 풀 (서브넷 + 볼륨타입 + 정책)
     → 사용자 그룹/계정 생성 → VM 할당 (풀 선택)
        → Nova: Zone의 Host에 배치
        → Neutron: Port 자동 생성 + IP 할당
        → Cinder: Volume 생성 (NetApp) → VM에 attach
        → Metadata 정책 → VM 하드웨어 구성 (모니터, USB 포트, SPICE 설정)

  4. 사용자 접속
     → SPICE 프로토콜로 VM 원격 접속
     → USB 정책에 따라 USB 장치 허용/차단
     → Metadata 정책에 따라 모니터 수, 해상도, 압축 방식 적용

  5. 포털 관리: 공지/FAQ/매뉴얼/다운로드/팝업 (user-portal 콘텐츠)
```

### 8단계: user-portal 서비스 시작

```
사용자: /{tntUrlId}/login → 1차+2차 인증 → 홈 → VDI 접속 (skbvdi://)
```

## 전체 요약

```
[인프라]  OpenStack 구축 (Host, Provider Network, NetApp, OS Image)
    ↓
[SA]     /initialized 마법사 → OpenStack → 로컬 DB 동기화
    ↓
[SA]     테넌트 + Zone 할당 → Flavor + 골든이미지 + 템플릿
    ↓
[SA]     상세 정책 생성 (예외NW, 접근차단, USB, Metadata, 블랙/화이트리스트 등)
    ↓
[SA]     인증 정책 + 가상PC 정책(허브) + 백업정책 + 볼륨타입/QoS + RBAC → TA 계정 생성
    ↓
[TA]     가상 네트워크 + 서브넷 + 라우터
    ↓
[TA]     가상PC 정책에서 상세 정책 선택 (접근차단/USB/Metadata/Proxy 등)
    ↓
[TA]     가상 PC 그룹 + 풀 → 사용자에게 VM 할당
    ↓
[OpenStack] VM 배치(Nova) + IP 할당(Neutron, SNAT N:1) + 디스크(Cinder/NetApp)
         + Metadata → VM 하드웨어 구성 (모니터/USB포트/SPICE)
    ↓
[사용자]  user-portal → SPICE로 VDI 접속 (USB/워터마크/접근차단 정책 적용)
```

## 네트워크 확인 사항 (실제 환경)

### Floating IP = 라우터 SNAT (N:1)

같은 테넌트 VM 2대가 동일한 외부 IP(`192.168.54.50`) 공유 확인. VPN 차단이 동료에게 전파되는 현상 실제 발생.

### 서브넷 DNS

서브넷의 DNS 1/DNS 2 설정 → 해당 서브넷 모든 VM이 이 DNS 서버 사용.

## 미확인 사항

- 초기 DB 데이터 삽입 방법 (SQL? Liquibase? 수동?)
- 테넌트 생성 시 OpenStack Project 자동 생성 여부 (TenantManagerController POST 미분석)
- 서비스 그룹(serviceGroup)의 역할과 테넌트와의 관계
- 마법사 Step 0 도메인 선택 시 테넌트 자동 생성의 정확한 동작
- 호스트 테넌트 모드(`HOST_TNT_YN = 'Y'`)의 정확한 물리적 격리 여부
- Floating IP 1:1 할당 vs SNAT N:1은 환경 설정에 따라 다를 수 있음
- Proxy 서버 정책(`proxy_type_cd`, P012 코드)의 실제 동작과 VDI 접속 경로에서의 역할
- 골든 이미지 레코드(CT_IMG_M) 최초 생성 경로 — admin-portal에 생성 UI 없음. 마법사 동기화? 백엔드 내부 로직? DB 직접 INSERT?
- 골든 이미지 생성용 전용 VM(CT_TNT_IMG_L.VM_ID)이 어떻게 할당/연결되는지
- Metadata 정책이 VM 생성 시 libvirt XML로 변환되는 정확한 경로 (app-ms-resource → plat-ms-vid4o → Nova 중 어디서 변환?)
