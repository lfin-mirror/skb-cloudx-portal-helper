# HA 실행 모드 — 자동/수동 동작 추측

테넌트 설정의 `HA 실행 모드 선택 (ha_auto_mng_yn)` 값에 따른 HA(Evacuate) 동작 차이를 코드 기반으로 정리.

## 추측 근거

1. `IndexDetail.vue:1894-1898` — TA 저장 시 host_tnt_yn에 따른 ha_auto_mng_yn 분기 (코드 검증)
2. `EvacuateService.java:72` — 자동 모드일 때 수동 evacuate 실행 차단 (코드 검증)
3. `EvacuateService.java:76-81` — 수동 evacuate 실행 조건: 호스트 `P003DWN` + evacuate 상태 정상 (코드 검증)
4. `VMController.java:196-198` — 별도 evacuate 엔드포인트 (자동 HA 트리거 경로로 추측)
5. `EvacuateHistory.vue:175` — 이력에서 `auto_mng_yn` Y→자동, N→수동 표시 (코드 검증)
6. 자동 HA의 호스트 장애 감지 메커니즘 (코드 미확인 — 추측)

## 기본값

| 조건 | `ha_auto_mng_yn` 기본값 | 근거 |
|------|----------------------|------|
| SA가 테넌트 저장 | 설정하지 않음 (SA 화면에 비노출) | `IndexDetail.vue:1889` — SA는 ha_auto_mng_yn 미전송 |
| TA + host_tnt_yn = 'Y' | TA가 직접 선택 (자동/수동) | `IndexDetail.vue:1894-1895` |
| TA + host_tnt_yn = 'N' | **강제로 'Y' (자동)** | `IndexDetail.vue:1896-1897` — "호스트 미사용 상태에서는 HA를 미노출 시키고 디폴트 값으로 설정" |

**N모드에서 자동 고정인 이유 (추측)**: N모드는 호스트를 테넌트가 직접 관리하지 않으므로 (플랫폼이 Zone 내 자동 배치), 장애 시에도 플랫폼이 자동으로 처리하는 게 자연스러움. 호스트를 특정할 수 없으니 수동 대피 대상을 선택하기도 어려움.

## 자동 모드 (`ha_auto_mng_yn = 'Y'`)

### 동작 (코드 검증 + 추측)

```
[호스트 장애 발생]
     │
     │ 호스트 상태 P003DWN 감지 (감지 메커니즘 미확인 — 추측)
     ▼
[자동 Evacuate 트리거]
     │
     │ VMController POST /v1/resource/vpcs/evacuate (추측)
     │   → vmService.getVmListByHostName() — 장애 호스트의 VM 목록 조회
     │   → vmActionService.evacInit() — evacuate 초기화 (evac_grp_id 생성)
     │   → vmActionService.evacuateVMAsync() — 비동기 VM 대피 실행
     ▼
[각 VM을 다른 호스트로 이동]
     │
     │ plat-ms-vid4o → Nova liveMigrate()
     ▼
[완료 / 에러]
     │ HA 이력에 auto_mng_yn = 'Y' (자동) 기록
```

### 수동 Evacuate 버튼 클릭 시 (코드 검증)

```java
// EvacuateService.java:72
if("Y".equals(detail.getHa_auto_mng_yn())) {
    throwMessageException("evc.message.002");  // 에러 — 자동 모드에서는 수동 실행 불가
}
```

자동 모드일 때 admin-portal에서 수동 HA 실행 버튼을 누르면 **서버에서 차단**. "자동 HA가 설정되어 있어 수동 실행 불가" 류의 에러 메시지 (추측).

## 수동 모드 (`ha_auto_mng_yn = 'N'`)

### 동작 (코드 검증)

```
[호스트 장애 발생]
     │
     │ 호스트 상태 P003DWN 감지
     │ → 자동 Evacuate 미실행 (수동 모드이므로)
     ▼
[관리자가 admin-portal에서 확인]
     │
     │ HA 실행 화면(EvacuateList)에서 P003DWN 상태 확인
     │ → HA 실행 버튼 클릭
     ▼
[EvacuateService.evacuate(host_id)]       ← POST /v1/evacuate/{host_id}
     │
     ├── 1. evacuateMapper.getHostInfo() — 호스트 상태 확인
     │      → host_sts_cd = 'P003DWN' (장애) 확인
     │      → evac_sts_cd 비어있음 (이전 evacuate 없음) 확인
     │      → ha_auto_mng_yn = 'N' (수동) 확인 — 'Y'면 에러
     │
     ├── 2. vmService.getVmListByHostName() — 장애 호스트의 VM 목록 조회
     │
     ├── 3. vmActionService.evacInit() — evacuate 초기화
     │      → autoYn = 'N' (수동) 세팅
     │
     └── 4. vmActionService.evacuateVMAsync() — 비동기 VM 대피 실행
            → 각 VM에 대해 plat-ms-vid4o → Nova liveMigrate()
            → HA 이력에 auto_mng_yn = 'N' (수동) 기록
```

### 수동 실행 조건 (코드 검증)

두 조건을 **모두** 만족해야 실행 가능:

| 조건 | 필드 | 값 | 의미 |
|------|------|-----|------|
| 호스트 장애 상태 | `host_sts_cd` | `P003DWN` | 호스트가 다운 상태여야 함 |
| 이전 evacuate 없음 | `evac_sts_cd` | 빈값 / null | 이미 진행 중인 evacuate가 없어야 함 |

+ `ha_auto_mng_yn = 'N'` (수동 모드) — 자동이면 서버에서 차단 (evc.message.002)

## 자동 vs 수동 비교

| 항목 | 자동 (`Y`) | 수동 (`N`) |
|------|-----------|-----------|
| 트리거 | 호스트 장애 감지 시 자동 실행 (추측) | 관리자가 직접 HA 실행 버튼 클릭 |
| 수동 실행 버튼 | **서버에서 차단** (evc.message.002) | 사용 가능 |
| host_tnt_yn = 'N'일 때 | **강제 자동** (프론트엔드에서 'Y' 고정) | 선택 불가 |
| host_tnt_yn = 'Y'일 때 | TA가 선택 가능 | TA가 선택 가능 |
| SA 화면에서 | **비노출** | **비노출** |
| 적합한 환경 (추측) | 무중단 운영 필수 환경 | 관리자가 대피 시점/대상을 직접 제어하고 싶은 환경 |
| HA 이력 표시 | `auto_mng_yn = 'Y'` → "자동" | `auto_mng_yn = 'N'` → "수동" |

## 자동 HA 트리거 메커니즘 (미확인)

코드에서 자동 HA가 **어떻게 트리거되는지**(호스트 장애 감지 → evacuate 자동 실행) 경로는 확인 안 됨.

가능한 시나리오 (추측):
1. **외부 모니터링 시스템** — Nagios/Zabbix 등이 호스트 down 감지 → app-ms-resource API 호출
2. **배치 스케줄러** — 주기적으로 호스트 상태 폴링, P003DWN 감지 시 evacuate 실행
3. **OpenStack Masakari** — OpenStack 공식 HA 서비스. 호스트 장애 시 자동 evacuate. 다만 CloudX 코드에서 Masakari 연동 흔적 미발견

`VMController`에 별도 evacuate 엔드포인트(`POST /v1/resource/vpcs/evacuate`)가 있는데, 이것이 자동 HA 트리거의 진입점일 가능성이 높음. `EvacuateController`의 수동 실행(`POST /v1/evacuate/{host_id}`)과 분리되어 있고, `VMController` 쪽은 `ha_auto_mng_yn` 체크 없이 바로 실행.

## 미확인 사항

- 자동 HA의 호스트 장애 감지 메커니즘 (폴링? 이벤트? 외부 시스템?)
- `VMController.evacuate()`와 `EvacuateController.evacuate()`의 정확한 역할 분담
- 자동 HA 실행 시 대상 호스트 선택 로직 (같은 Zone 내 다른 호스트? 가장 여유 있는 호스트?)
- `evc.message.001`, `evc.message.002`, `evc.message.003`의 정확한 에러 메시지 텍스트
- SA가 ha_auto_mng_yn을 설정하지 않는 이유 (DB 기본값이 있는지?)
