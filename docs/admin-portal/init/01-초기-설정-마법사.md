# 초기 설정 마법사 (`/initialized`)

SA 전용. 플랫폼 최초 구축 시 OpenStack 인프라를 CloudX에 동기화하는 설정 마법사.

- 경로: `/initialized`
- 컴포넌트: `views/initialized/index.vue` (1379줄)
- 라우터: `router/index.js` line 65-67 (`hidden: true`)
- 스토어: `store/modules/init.js`

## 진입 조건

SA 로그인 후 `nextRouter()`에서 분기:

```
login/index.vue nextRouter() (line 980):
  if (user.acct_sts_cd === 'U013IN') {
    path = '/initialized'
  }
```

`acct_sts_cd === 'U013IN'` (초기화 필요) 상태일 때만 자동 리다이렉트. 정상 상태(`U013NR`)에서는 대시보드로 이동.

## 실행 모드

### 자동 모드 (`selectedOption='A'`)

"시작" 버튼 클릭 → 전체 단계를 순차 자동 실행.

```
1. beforeStepHandller() 호출
2. stepHandllerAuto() 시작
3. 도메인(step 0): DomainCreate 모달 → 도메인이 이미 있으면 자동 스킵
4. 나머지 단계(step 1~8): 각 단계마다
   → stepStart(): POST /v1/resource/init (백엔드 작업 시작)
   → 1.5초 간격으로 폴링: GET /v1/resource/init (상태 확인)
   → 작업 완료(J008E) 또는 경고(J008W) → stepIncrement()로 다음 단계
   → 작업 실패(J008F) → 정지, 에러 모달 표시
```

### 수동 모드 (`selectedOption='M'`)

각 단계를 사용자가 개별 트리거. 폴링 메커니즘은 동일.

## 단계별 상세

### Step 0: 도메인 (수동)

`DomainCreate.vue` 모달 표시.

```
1. OpenStack에서 도메인 목록 조회
2. 도메인이 이미 존재 → domainSuccessModal() → 자동 스킵
3. 도메인 없음 → 사용자가 선택 또는 생성
4. 도메인 선택 시 테넌트 자동 생성 동반
```

### Steps 1~8: 자동 동기화

각 단계의 공통 동작:

```
1. index.vue가 dataSwitch[step].trigger = true 설정
2. 해당 컴포넌트가 trigger watch → API 호출하여 데이터 조회
3. 조회 성공 → 'XxxClearCallback' emit
4. 조회 실패 → 'errorCallback' emit
```

`code=init` 파라미터: 백엔드에서 초기 설정 전용 로직 분기에 사용. OpenStack에서 리소스를 읽어와 로컬 DB에 동기화하는 작업 수행.

## 폴링 메커니즘

### 백엔드 작업 상태 코드

| 코드 | 의미 | 마법사 동작 |
|------|------|-----------|
| `J008I` | 작업 중 (In progress) | 폴링 계속 |
| `J008E` | 완료 (End) | 폴링 중지, 다음 단계 진행 |
| `J008W` | 경고 (Warning) | 폴링 중지, 다음 단계 진행 (경고 알림) |
| `J008F` | 실패 (Failure) | 폴링 중지, 에러 모달 표시, 재시도 대기 |

### 폴링 API

```
시작: POST /v1/resource/init
  → 백엔드에서 현재 단계의 동기화 작업 시작
  → init 스토어의 stepJobData.code = 'J008I' 설정

폴링: GET /v1/resource/init
  → 1.5초 간격 (setInterval)
  → 응답에서 sub_job_id가 현재 stepStatus와 매칭되는 항목의 job_sts_cd 확인
  → J008E/J008W → clearInterval + stepIncrement()
  → J008F → clearInterval + isRetry = true + 에러 알림
```

## Vuex init 스토어 사용

| 액션 | 역할 |
|------|------|
| `stepStart` | `POST /v1/resource/init` → 백엔드 작업 시작, 상태를 J008I로 설정 |
| `stepApiStatusAsync` | `GET /v1/resource/init` → 현재 작업 상태 폴링 |
| `stepIncrement` | 단계 카운터 증가 (`nowStepStatus += count`), stepFinish 호출 |
| `stepFinish` | 성공/경고 알림 표시, `finishStepStatus` 업데이트 |

## 완료 처리

Step 8 (Proxy) 완료 후 `update()` 메서드 실행:

```
1. GET /v1/user/accounts/{acctId}
   → 현재 사용자 계정 조회

2. 계정 정보 수정:
   - grp_typ_cd: 'U001SUP' (Super Admin 확정)
   - acct_sts_cd: 'U013NR' (정상 상태로 변경)
   - acct_vlid_stt_dt / end_dt: 하이픈 제거

3. PUT /v1/user/accounts/{acctId}/admin
   → 수정된 계정 정보 저장

4. DELETE /v1/user/accounts/u013in
   → U013IN 마커 삭제

5. alert("모든 초기화 작업이 완료되었습니다")

6. store.dispatch('LogOut')
   → 로그아웃 후 /login으로 리다이렉트 (location.href)
```

재로그인 시 `acct_sts_cd`가 `U013NR`이므로 `nextRouter()`에서 대시보드로 이동.

## 초기 설정 완료 후 재접근

- `/initialized` 라우트는 `hidden: true`지만 직접 URL 접근 가능
- `permission.js`에 별도 차단 로직 없음
- 다만 `acct_sts_cd`가 `U013NR`이면 로그인 후 자동 리다이렉트 안 됨
- 직접 접근 시 백엔드에 초기화 데이터 없으므로 빈 화면

## 초기 설정이 하는 일 요약

```
OpenStack 인프라 (이미 구축됨)
  ↓ /initialized 마법사
  ↓ code=init 파라미터로 백엔드 동기화 작업 트리거
CloudX 로컬 DB에 메타데이터 동기화:
  - 도메인 → 도메인/테넌트 정보
  - 호스트 → Compute Node 목록
  - Zone → Host Aggregate/AZ 매핑
  - 이미지 → Glance 이미지 목록
  - Flavor → 가상 PC 사양
  - 네트워크 → Provider Network
  - 서브넷 → 서브넷 CIDR/DHCP 설정
  - 볼륨 → Volume Type/Backend
  - Proxy → L4/L7 Proxy 서버 정보
```

이 동기화가 완료되어야 admin-portal의 "시스템 자원" 메뉴에서 로컬 DB 기반 목록 조회가 가능해짐.

## 초기 설정의 위치 — 전체 구축 흐름에서 (추측)

아래는 코드 분석 기반 추측. 실제 구축 가이드 미확인 상태.

### 논리 ↔ 물리 연결 구조

```
논리적 구조 (Keystone)          연결 지점              물리적 구조 (Nova)
─────────────────────    ─────────────────    ─────────────────────
Domain
  └── Tenant ──────────── CV_TNT_M.ZONE_NM ──── Zone (Availability Zone)
        │                                         └── Host Aggregate (=Zone)
        │                                              ├── Host (compute-01)
        │                                              └── Host (compute-02)
        │                                                    │
        └── VM (가상 PC) ─── 배치(스케줄링) ──────────────────┘
```

Keystone(누구 것?) → `CV_TNT_M`(어떤 Zone?) → Nova(어디서 실행?) → Neutron(어떻게 통신?) → Cinder/NAS(어디에 저장?)

OpenStack에는 테넌트 → Zone 연결이 공식적으로 없음. CloudX가 `CV_TNT_M` 테이블로 자체 구현한 연결.

### 추측되는 구축 순서

```
1단계: OpenStack 인프라 구축 (CloudX 이전)
  → Keystone: Domain 생성
  → Nova: Compute Node 설치, 하이퍼바이저 구성
  → Neutron: Provider Network 구성
  → Cinder: Storage Backend(NetApp 등) 연결
  → Glance: OS 이미지 등록

2단계: CloudX 마이크로서비스 배포
  → plat-ms-vid4o: OpenStack 접속 정보 설정
  → app-ms-resource: DB 스키마 생성, vid4o.url 설정
  → app-ms-operation: 정책/운영 DB 스키마 생성
  → admin-portal / user-portal 배포

3단계: 초기 데이터 세팅 (DB 직접 삽입 추측)
  → SA 초기 계정 INSERT (acct_sts_cd = 'U013IN')
  → 공통 코드/메뉴 트리 데이터 INSERT

4단계: SA 첫 로그인 → /initialized 마법사 ← 이 문서의 내용
  → OpenStack 인프라를 로컬 DB에 동기화
  → 도메인 선택 → 호스트/Zone/이미지/Flavor/네트워크/서브넷/볼륨/Proxy

5단계: 테넌트 생성 및 Zone 할당
  → SA가 테넌트 관리(A0802)에서 테넌트 생성
  → CV_TNT_M에 테넌트 + ZONE_NM 매핑 저장
  → 이때 논리(테넌트) ↔ 물리(Zone) 연결 확립

6단계: TA 계정 생성 → 서비스 운영
  → SA가 TA 계정 생성, 테넌트 할당
  → TA 로그인 → 자기 테넌트의 Zone/호스트만 조회 가능
  → 가상 PC 그룹/풀 생성 → VM 배치 → user-portal 서비스 시작
```

### 이 마법사가 하는 일의 본질 (추측)

**OpenStack은 이미 구축된 인프라.** 이 마법사는 "OpenStack에 뭐가 있는지 읽어와서 CloudX DB에 기록"하는 동기화 작업. 이후 admin-portal의 목록 조회는 OpenStack을 매번 호출하지 않고 로컬 DB에서 빠르게 처리. 생성/수정/삭제 시에만 OpenStack API 호출 + 로컬 DB 동기화.
