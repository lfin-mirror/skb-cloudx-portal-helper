# Glance — 이미지 서비스

admin-portal 메뉴: 템플릿 > 베이스 이미지 (A0903/T0903), 골든 이미지 생성 (A090302)

## OpenStack Glance

VM을 부팅하기 위한 **디스크 이미지 파일을 저장·관리**하는 서비스. admin-portal에서 "베이스 이미지"로 표시되는 것의 실체.

## 이미지가 뭔가

```
이미지 = "OS가 설치된 가상 디스크 파일"

윈도우 설치 USB를 떠올리면 됨:
  USB (이미지) → PC에 꽂고 부팅 → 윈도우 실행
  이미지 (Glance) → VM에 적용하고 부팅 → 가상 PC 실행
```

이미지 파일 형식: qcow2, raw, vmdk 등. 수 GB ~ 수십 GB.

## 이미지의 저장 위치

```
admin-portal에서 보이는 "베이스 이미지 목록"
  └── 로컬 DB → 메타데이터만 (이미지명, ID, OS 유형, 생성일)

실제 이미지 파일
  └── OpenStack Glance → 스토리지 백엔드
        ├── Ceph (분산 스토리지)
        ├── NFS (네트워크 파일 시스템)
        └── 로컬 파일시스템 (/var/lib/glance/images/)
```

## VM 생성 시 이미지 사용 흐름

```
1. [TA] 사용자에게 VM 할당 → 풀의 템플릿에서 img_id 가져옴
     템플릿: "Win10-Standard" → img_id: "abc-123-def"

2. [app-ms-resource] → plat-ms-vid4o → Nova API 호출
     "이미지 abc-123-def, Flavor xyz로 VM 생성해줘"

3. [Nova] Glance에 이미지 요청
     "이미지 abc-123-def 줘"

4. [Glance] 스토리지 백엔드에서 이미지 바이너리 읽기
     → 수 GB 파일을 호스트(compute-01)로 전송

5. [Nova] 호스트에 이미지 캐시 → 루트 디스크 생성 → VM 부팅
     → 같은 이미지로 두 번째 VM 생성 시 캐시 사용 (빠름)
```

### 이미지 캐시

같은 호스트에서 같은 이미지로 VM을 여러 대 만들면, 첫 번째만 Glance에서 다운로드하고 이후는 호스트 로컬 캐시 사용. 골든 이미지처럼 많은 VM을 찍어내는 경우 효과적.

## 이미지 종류

| 종류 | 설명 | 생성 방법 |
|------|------|----------|
| 베이스 이미지 | 순수 OS 이미지 (소프트웨어 미설치) | 인프라팀이 Glance에 직접 업로드 |
| 골든 이미지 | OS + 소프트웨어가 미리 설치된 이미지 | admin-portal에서 생성 (아래 참고) |

## 골든 이미지 생성 과정

```
1. [SA] 골든 이미지 생성 화면 (A090302)
     → 생성용 VM 목록에서 VM 선택

2. [SA] VM에 VDI 접속
     → Windows 설치, Office 설치, 보안 소프트웨어 설치 등

3. [SA] "배포" 클릭
     → VM의 현재 루트 디스크 상태를 Glance에 새 이미지로 저장
     → Nova → Glance: "이 VM의 디스크를 이미지로 만들어줘"
     → Glance가 스토리지 백엔드에 새 이미지 파일 생성

4. [SA/TA] 베이스 이미지 목록에 새 골든 이미지 표시
     → 이후 템플릿에서 이 골든 이미지 선택 가능
     → 이 템플릿으로 VM 생성 → 매번 소프트웨어 설치 불필요
```

### 골든 이미지가 필요한 이유

```
골든 이미지 없이:
  VM 100대 생성 → 각 VM에 접속해서 Office, 보안 SW, 사내 앱 설치
  → 100번 반복 (비현실적)

골든 이미지 사용:
  VM 1대에 모든 소프트웨어 설치 → 골든 이미지로 저장
  → 이 이미지로 VM 100대 찍어냄 → 전부 소프트웨어 설치 완료 상태
```

## 각 리소스의 저장 위치 비교

| 리소스 | 메타데이터 | 실체(바이너리/설정) |
|--------|----------|-------------------|
| 베이스 이미지 | 로컬 DB (이름, ID) | **Glance → 스토리지 백엔드** (디스크 이미지 파일) |
| Flavor (가상 PC 사양) | 로컬 DB (이름, 스펙) | **Nova** (스케줄링 설정값, 파일 아님) |
| 템플릿 | 로컬 DB에만 존재 | **CloudX 자체 개념** (Flavor ID + Image ID 조합) |
| Volume (디스크) | 로컬 DB (이름, 상태) | **Cinder → NetApp** (블록 디스크) |

템플릿은 OpenStack에 존재하지 않는 CloudX 자체 개념. OpenStack에서는 VM 생성 시 이미지와 Flavor를 각각 지정하는데, CloudX가 이 두 가지를 미리 조합해놓은 것이 템플릿.

## Glance 이미지와 Cinder 스냅샷/볼륨의 관계

Glance 이미지와 Cinder 볼륨/스냅샷은 다른 개념이지만, OpenStack에서는 서로 변환 가능.

### 변환 케이스

```
케이스 1: 이미지 → 볼륨 (VM 생성 시)
  Glance 이미지로 Cinder 볼륨 생성 → VM의 루트 디스크로 사용
  → 가장 흔한 케이스. VM 생성 = 이미지에서 볼륨 복제

케이스 2: 볼륨 → 스냅샷 (백업)
  운영 중인 VM 디스크의 특정 시점 상태 저장
  → user-portal "스냅샷 및 복원" 기능

케이스 3: 스냅샷 → 볼륨 (복원)
  스냅샷 시점으로 새 볼륨 생성
  → 테스트 환경 생성, 데이터 복구

케이스 4: 스냅샷 → Glance 이미지 (이미지 등록)
  스냅샷 상태를 VM 생성용 이미지로 등록
  → 특정 시점의 디스크를 템플릿으로 재사용

케이스 5: 볼륨 → Glance 이미지 (골든 이미지 생성)
  운영 중이던 디스크를 이미지로 만들어 재사용
  → admin-portal 골든 이미지 "배포" = 이 케이스
```

```
시각화:

  Glance 이미지 ──(케이스1)──→ Cinder 볼륨
       ↑                          │  ↑
       │                   (케이스2) │  │ (케이스3)
  (케이스4,5)                      ↓  │
       │                     Cinder 스냅샷
       └──────────────────────────┘
```

### Glance + Cinder 백엔드 통합

일반적으로 Glance 이미지는 파일 형태(Swift, Ceph RGW, 파일시스템 등)로 저장. 하지만 Glance를 **Cinder 백엔드로 구성**하면 구조가 달라짐:

```
일반 구성:
  Glance 이미지 → 파일시스템/Ceph (파일 형태)
  Cinder 볼륨 → NetApp/Ceph (블록 형태)
  → 이미지→볼륨 변환 시 데이터 전체 복사 필요 (느림)

Cinder 백엔드 구성 (use_cinder_backend_for_image_upload = True):
  Glance 이미지 → Cinder 볼륨으로 저장 (블록 형태)
  Cinder 볼륨 → 같은 스토리지
  → 논리적으로는 "Glance 이미지"지만 물리적으로는 Cinder 볼륨
  → Glance 이미지 1개 = Cinder 볼륨 1개
```

### 스토리지 레벨 오프로딩 (Offload)

Cinder 백엔드 구성의 핵심 이점: **데이터 복사 없이 클론 생성.**

```
일반 구성에서 이미지 → 볼륨:
  1. Glance에서 이미지 파일 읽기 (10GB)
  2. 네트워크로 전송
  3. Cinder가 새 볼륨에 데이터 쓰기
  → 10GB 풀 복사. 수 분 소요.

Cinder 백엔드에서 이미지 → 볼륨:
  1. "이 볼륨을 클론해줘" (스토리지 레벨 명령)
  2. 스토리지(NetApp/Ceph)가 내부적으로 클론
  → 데이터 전송 없음. 수 초 소요.
```

### Copy-on-Write (CoW)

클론 시 실제 데이터를 복사하지 않고 **원본을 참조**만 하다가, 수정이 발생한 블록만 새로 기록하는 방식.

```
골든 이미지 (10GB) 기반으로 VM 100대 생성:

풀 복사 방식:
  10GB × 100 = 1TB 스토리지 사용
  각 VM 생성에 수 분

CoW 방식:
  원본 10GB + 변경분만 추가 (VM당 수 MB~수 GB)
  총 사용량: 10GB + α (훨씬 적음)
  각 VM 생성에 수 초 (클론은 메타데이터 조작만)
```

```
시각화:

  골든 이미지 (원본, 10GB)
    ├── VM-001 볼륨 (CoW: 원본 참조 + 변경분 200MB)
    ├── VM-002 볼륨 (CoW: 원본 참조 + 변경분 150MB)
    ├── VM-003 볼륨 (CoW: 원본 참조 + 변경분 300MB)
    └── ...

  VM-001이 C:\에 파일 저장 → 변경된 블록만 VM-001 전용 영역에 기록
  원본(골든 이미지)은 불변 유지
```

VDI 환경에서 CoW가 특히 효과적인 이유: 같은 골든 이미지로 수백 대의 VM을 찍어내는데, 대부분의 데이터(OS, 기본 소프트웨어)는 동일하고 사용자별 변경분만 다르기 때문.

NetApp은 FlexClone 기술로 CoW 클론 지원. CloudX가 NetApp을 기본 백엔드로 사용하는 이유 중 하나.

## plat-ms-vid4o API

| 메서드 | 경로 | OpenStack API | 용도 |
|--------|------|---------------|------|
| GET | `/image/images` | Glance `GET /v2/images` | 이미지 목록 |
| GET | `/image/images/{imageId}` | Glance `GET /v2/images/{id}` | 이미지 상세 |
| POST | `/image/images` | Glance `POST /v2/images` | 이미지 등록 |
| DELETE | `/image/images/{imageId}` | Glance `DELETE /v2/images/{id}` | 이미지 삭제 |

## 마법사에서의 동기화

`/initialized` 마법사 Step 3에서 이미지 동기화:

```
GET /v1/resource/images?code=init
  → Glance에서 이미지 목록 읽어와 로컬 DB에 저장
  → 이후 admin-portal에서 로컬 DB 기반 목록 조회
```
