# 릴리스 기반 문서 업데이트 규칙 (경로 B)

문서 없이 코드만 머지된 경우, 업데이트 내역을 받아서 소스코드를 분석하고 문서를 업데이트하는 프로세스.

## 발생 시나리오

- 다른 팀이 개발한 기능이 v2에 머지됨
- 핫픽스가 문서 업데이트 없이 반영됨
- 과거 버전에서 쌓인 변경사항을 일괄 문서화

## 프로세스

### 1. 업데이트 내역 수신

입력 형식:

```markdown
# v2.3.0 업데이트 내역

| JIRA | 유형 | 설명 | 영향 화면 |
|------|------|------|----------|
| IAADEV-1234 | 기능 추가 | 가상PC 네이밍 정책 | 정책 > 상세 정책 |
| IAADEV-5678 | 변경 | 워터마크 옵션 추가 | 정책 > 가상 PC 정책 |
```

영향 화면을 모르는 경우 JIRA 설명 + git diff로 추적.

### 2. 변경 범위 파악

```
git diff v2.2.9..v2.3.0 -- '*.vue' '*.js'
```

- JIRA 티켓에서 영향받는 화면/기능 식별
- 변경된 Vue 컴포넌트 파일 추출

### 3. 소스코드 분석

변경된 컴포넌트를 읽고:
- 새 필드/버튼/API/분기 조건 파악
- 기존 화면 문서와 비교

### 4. 문서 업데이트

- 기존 문서 수정: 새 필드 추가, 삭제된 필드 제거, API 변경 반영
- 신규 화면: 새 문서 생성 (frontmatter + 본문)
- frontmatter `version` 업데이트
- `## 변경 이력`에 JIRA 티켓 기록
- api/ 명세도 변경 시 함께 업데이트

**버그픽스는 화면 문서에 반영하지 않는다.** UI 동작 수정, 입력 오류 수정, 스타일 보정 등 기존 화면의 구조·필드·API를 변경하지 않는 버그픽스는 화면 문서 수정 대상에서 제외. changelog에만 포함.

### 5. changelog 작성

`changelog/{버전}.md`에 변경 요약 작성.

## 화면 문서 찾기

변경된 컴포넌트 → 화면 문서 매칭 방법:
- frontmatter의 `component` / `component_sa` / `component_ta` 필드로 검색
- frontmatter의 `menu_id`로 검색
- frontmatter의 `api_endpoints`로 검색

## 자동화

상세 파이프라인(Phase 0~4, 에이전트 분리, api-quality 연계, 사용자 확인 게이트)은 `.claude/skills/docs-release/skill.md` 참조.

## AI 실행 명령 예시

```
소스코드를 분석해서 화면 문서를 업데이트해줘."
```
